App-level configuration

The title found in the center of the page header should be changed in src/main/webapp/scripts/app.js. Specifically, the line that reads 

    .constant('appTitle', 'Polaris Angular App')          <-- changes the title in the center of the page header
    .constant('roleUrl', '/url/path/to/role')             <-- relative url where the user's role is retrieved from
    .constant('attributeUrl', '/url/path/to/attributes')  <-- relative url where the application attributes are retrieved from
    .constant('menuUrl', '/url/path/to/menu')             <-- relative url where the application attributes are retrieved from
    .constant('appTabs', [
        {id: 'tab1', desc: 'Page Tab 1', url: 'url', isActive: 'active'},
        {id: 'tab2', desc: 'Page Tab 2', url: 'url', isActive: ''},
        {id: 'tab3', desc: 'Page Tab 3', url: 'url', isActive: ''},
        {id: 'tab4', desc: 'Page Tab 4', url: 'url', isActive: ''}
    ])

Example of beginning development

Typical Angular Single Page App (SPA) development at Polaris is highly data dependent. To get started most developers will need to retrieve data from some RESTful endpoint and produce that data on page. Below are the high-level steps to get started:

  1. Create an Angular $resource within the src/main/webapp/scripts/resources directory. If you don't have a real RESTful endpoint to retrieve data from, you can mock your data in the return of the Angular $resource you'll create. Just make sure it returns a $promise rather than the actual object. Then, whatever code is dependent upon your resource will not have to be refactored once you introduce a real response.
  2. Determine whether or not the data should be retrieved once per user session or retrieved every time the data is needed. If it should be retrieved once and cached locally you can implement a service in the src/main/webapp/scripts/services directory that can manages that functionality.
  3. Inject your new $resource or your new service (depending on caching requirements) in to a page controller so the data will be presented to the user when the controller is accessed.
